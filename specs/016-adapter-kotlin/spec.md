# Feature Specification: Kotlin Adapter — Jetpack Compose Codegen

**Spec ID**: 016-adapter-kotlin

**Created**: 2026-05-28

**Status**: Implemented

**Input**: Third native adapter, completing the platform triad alongside Flutter and Swift.
Android doesn't expose a JS runtime, so this is **build-time codegen**: turn a Polymorph theme
into a self-contained `.kt` file that the consumer's Gradle build compiles. No Android SDK
required at codegen time — the package emits text.

---

## Overview

`@polymorph/adapter-kotlin` exports `transformToKotlin(theme, options?)` and per-type Kotlin
value converters; the `polymorph` CLI's existing `transform` command gains `--target kotlin`.

The emitted file declares two helper data classes (`PolymorphTextStyle`, `PolymorphShadow`)
inlined at the top — so the consumer's Android app has **no external Polymorph dependency** —
followed by a top-level `object` (singleton namespace) with one `val` per resolved semantic +
component token.

Why a top-level `object` instead of a `class`/`companion object`? Kotlin's `object` declares a
singleton namespace — exactly the right primitive for "static constants only" — and it matches
common community conventions (e.g. `object Constants { ... }`).

## Clarifications

### Session 2026-05-28

- Q: Runtime provider vs. build-time codegen? → A: **Codegen only** for v1, same as Flutter
  and Swift. Compose theming is idiomatically compile-time (`val` constants,
  `CompositionLocal`, theme objects); a runtime DTCG parser would force JSON parsing at app
  start. Build-time emits pure Kotlin with no Polymorph dependency on the consumer side.
- Q: Helper data classes in a runtime package or inlined? → A: **Inlined**. Avoids forcing the
  consumer to add a Maven dependency just to use the constants.
- Q: How to represent typography? → A: Compose's `TextStyle` requires a `FontFamily` built
  from `Font` resources — those live on the consumer side. Emit a `PolymorphTextStyle` data
  class with `fontFamily: String` + the rest as Compose types (`TextUnit`, `FontWeight`,
  `Float` lineHeight multiplier, `TextUnit` letterSpacing). Consumers bridge to `TextStyle`
  via a one-line `@Composable` extension (sample in README).
- Q: How to represent shadows? → A: Compose has no first-class shadow data type; the `.shadow`
  modifier takes raw `Dp` + `Color`. Emit `List<PolymorphShadow>` (custom data class with
  `color/x/y/radius`) so single-shadow and multi-shadow tokens are uniform.
- Q: Durations in ms or `Duration`? → A: **Kotlin `Int` milliseconds**. Compose's `tween()`
  takes `durationMillis: Int`, so this maps directly. `kotlin.time.Duration` would force every
  call site to convert.
- Q: Cubic-bezier as a tuple or Compose's easing type? → A: **`CubicBezierEasing(...)`** from
  `androidx.compose.animation.core` — it's a real Compose type that drops directly into
  `tween(easing = …)`.
- Q: Colors — `Color(0xFFRRGGBB)` or channel-based? → A: **Hex int constructor**, identical to
  Compose's standard idiom. `Color(0xFFRRGGBB)` is `@Stable` and `inline` so there's no
  per-frame allocation cost. Alpha forced to `FF` in v1.
- Q: Package declaration? → A: **Default `polymorph.theme`**, overridable via the programmatic
  `packageName` option. The CLI doesn't expose a flag yet — most apps will rename via their
  build's source set, and editing the first line of the file is trivial.
- Q: Test strategy? → A: **Committed Kotlin goldens** (`aurora_{light,dark}.kt`,
  `borealis_{light,dark}.kt`) regenerated by `pnpm --filter @polymorph/adapter-kotlin
  update-goldens` and asserted byte-equal; per-converter unit tests cover edges.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — A Compose app consumes a Polymorph theme (Priority: P1)

A vendor authors `aurora.tokens.json`, runs `polymorph transform aurora.tokens.json --target
kotlin --mode light --class AuroraThemeLight --output
app/src/main/java/polymorph/theme/AuroraThemeLight.kt`, and imports the resulting object:

```kotlin
import polymorph.theme.AuroraThemeLight

@Composable
fun Greeting() {
  Text(
    "Hello",
    color = AuroraThemeLight.colorTextBody,
    fontSize = AuroraThemeLight.typographyBody.fontSize,
    fontWeight = AuroraThemeLight.typographyBody.fontWeight,
    modifier = Modifier
      .padding(AuroraThemeLight.spaceMd)
      .background(AuroraThemeLight.colorSurfaceRaised),
  )
}
```

**Independent Test**: `tests/codegen.test.ts` — `transformToKotlin(theme, { mode, objectName })`
on the two bank fixtures across `light`/`dark` matches the four committed goldens byte-for-byte.

### User Story 2 — CI catches generator drift (Priority: P1)

`pnpm --filter @polymorph/adapter-kotlin run update-goldens` regenerates the four `.kt` files.
CI's drift guard re-runs the same script (after building the adapter so `tsx` can resolve
`@polymorph/core` to its `dist/`) and fails on any diff.

### User Story 3 — CLI integration (Priority: P2)

The existing `transform` command grows a third target. Same validate-first behaviour, same
exit codes (`0` success, `1` invalid theme, `2` usage).

### Edge Cases

- `currentColor` / unparseable colors → `colorToKotlin` returns `null`; the emitter skips that
  constant rather than emitting invalid Kotlin.
- `rem` dimensions multiplied by 16 (Compose has no rem; matches the spec's default root size).
- Typography `fontWeight` rounded to the nearest of `100..900` and mapped to Compose's
  `FontWeight.W100`–`FontWeight.W900`.
- Typography `fontSize` and `letterSpacing` emit as `TextUnit` via `.sp`; dimensional space /
  radius tokens emit as `Dp` via `.dp`. The pair is split so Compose's strong typing isn't
  fighting the generator.
- Shadow `inset` is dropped with an inline `/* inset not supported by Compose shadow */`
  comment — Compose's shadow modifier has no inset mode.
- The `--class` flag re-maps to `objectName` for this target (Kotlin objects are the namespace).

## Requirements *(mandatory)*

- **FR-001**: `transformToKotlin(theme, options)` MUST validate and resolve the theme via
  `@polymorph/core` before emitting; invalid themes throw the same errors `resolveTheme`
  throws.
- **FR-002**: Output MUST be a single `.kt` file containing the two helper data classes
  followed by one `object` namespace with `val`s per resolved semantic + component token,
  grouped under `// region … // endregion` banners.
- **FR-003**: The helper data classes (`PolymorphTextStyle`, `PolymorphShadow`) MUST be
  inlined in the generated file — no external Polymorph runtime dependency allowed.
- **FR-004**: Colors MUST accept any CSS Color 4 form via `@polymorph/core.parseColor` and emit
  `Color(0xFFRRGGBB)` (alpha forced to FF in v1; non-sRGB converted to sRGB).
- **FR-005**: Dimensions in `rem` MUST be multiplied by 16; spatial dimensions emit as Compose
  `Dp` (`16.0f.dp`); typography metrics emit as `TextUnit` (`16.0f.sp`).
- **FR-006**: Durations MUST be emitted as Kotlin `Int` milliseconds (`220ms` → `220`,
  `1s` → `1000`) — direct input to Compose's `tween(durationMillis = …)`.
- **FR-007**: `componentPropKotlinName(role, property)` MUST flatten dotted roles + camelCased
  properties into one camelCase identifier (`button.primary` + `background` →
  `buttonPrimaryBackground`).
- **FR-008**: The CLI's `transform` command MUST accept `--target kotlin` (alongside `dart`
  and `swift`); the `--class` flag is reused to name the Kotlin `object`.
- **FR-009**: Output MUST be deterministic for a given theme + options — manifest iteration
  order, fixed number formatting (always with the `f` suffix for `Float`), no timestamps.

## Assumptions

- v1 is codegen-only; no in-app DTCG runtime. The converters are reusable for a future
  Maven-distributed runtime if one materializes.
- Jetpack Compose is the primary surface. View-system Android consumers can bridge `Color` via
  `.toArgb()` and read `Dp` values through `LocalDensity`.
- The mock-bank themes are the de facto fixture surface. Adding new component-token roles to a
  bank requires regenerating goldens (caught by the drift guard).
- The default `package polymorph.theme` declaration is editable — consumers either set their
  own via the programmatic `packageName` option or rename via Gradle source-set roots.
