# Feature Specification: Flutter Adapter — Dart Codegen

**Spec ID**: 014-adapter-flutter

**Created**: 2026-05-28

**Status**: Implemented

**Input**: First non-web adapter. Flutter doesn't have a JS runtime we can ship a
context-provider into, so the adapter is **build-time codegen**: turn a Polymorph theme into a
self-contained `.dart` file that the consumer's Flutter build compiles. No Dart toolchain
required at codegen time — output is text.

---

## Overview

`@polymorph/adapter-flutter` exports `transformToDart(theme, options?)` and per-type Dart value
converters; the `polymorph` CLI gains a `transform --target dart` command that wires the
adapter to the existing validate/resolve pipeline.

For each resolved semantic + component token, the generated class exposes a `static const` of
the appropriate Dart type (`Color`, `double`, `TextStyle`, `Duration`, `Cubic`,
`List<BoxShadow>`). It also emits an opinionated `static ThemeData buildThemeData()` factory
that maps the most common semantic tokens onto Material's `ThemeData` / `ColorScheme` /
`TextTheme` — consumers can call it directly or cherry-pick constants.

## Clarifications

### Session 2026-05-28

- Q: Runtime provider vs. build-time codegen? → A: **Codegen only** for v1. Flutter's idiomatic
  theming is compile-time (`static const`, `ThemeData`); a runtime DTCG parser would force JSON
  parsing + reflection at app start and lose `const` constructor benefits. Build-time emits
  pure Dart with no Polymorph dependency on the consumer side.
- Q: One file or one per token group? → A: **One file, one class.** Easier to import; sections
  are delimited by comment banners (`// ---- Colors ----`) so it's still readable.
- Q: How to handle alpha and non-sRGB colors? → A: Force alpha to `FF` in v1 (no token in the
  banks uses semi-transparent; revisit when one does). All CSS Color 4 forms (`oklch`, `lab`,
  `display-p3`, etc.) flow through `@polymorph/core.parseColor` to clamped sRGB.
- Q: `rem` dimensions? → A: Multiply by 16 (Flutter has no rem; sticking with the spec's
  default root size keeps it predictable).
- Q: Shadow `inset`? → A: Not representable on `BoxShadow`. Emit a `// inset: unsupported in
  Flutter BoxShadow` comment in the list literal and skip that shadow.
- Q: Where do component-token names land? → A: `componentPropDartName(role, prop)` →
  camelCased flat identifier, e.g. `button.primary` + `background` → `buttonPrimaryBackground`,
  matching the Web adapter's CSS variable naming shape but in Dart identifier style.
- Q: Test strategy? → A: **Committed Dart goldens** (`aurora_{light,dark}.dart`,
  `borealis_{light,dark}.dart`) regenerated by `pnpm --filter @polymorph/adapter-flutter
  update-goldens` and asserted byte-equal in CI; per-converter unit tests cover edges.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — A Flutter app consumes a Polymorph theme (Priority: P1)

A vendor authors `aurora.tokens.json`, runs `polymorph transform aurora.tokens.json --target
dart --mode light --class AuroraThemeLight --output lib/polymorph_theme.dart`, and imports the
resulting class:

```dart
import 'package:flutter/material.dart';
import 'polymorph_theme.dart';

MaterialApp(theme: AuroraThemeLight.buildThemeData(), home: …);
// Or grab individual tokens:
Container(color: AuroraThemeLight.colorSurfaceRaised);
```

**Independent Test**: `tests/codegen.test.ts` — `transformToDart(theme, { mode, className })`
on the two bank fixtures across `light`/`dark` matches the four committed goldens byte-for-byte.

### User Story 2 — CI catches generator drift (Priority: P1)

`pnpm --filter @polymorph/adapter-flutter run update-goldens` regenerates the four `.dart`
files from the bank token sources. The CI drift guard re-runs the same script on every PR; a
non-empty `git diff` fails the build. Same pattern as the spec types/schemas and bank themes.

**Independent Test**: GitHub Actions `verify` job step "Verify generated artifacts are in sync"
includes the goldens regeneration.

### User Story 3 — CLI integration (Priority: P2)

`polymorph transform <file> --target dart [--mode <mode>] [--class <ClassName>] [--output
<path>]` validates the theme first, exits `1` on schema errors, exits `2` on missing/unknown
target, otherwise emits to stdout or writes to `--output`.

**Independent Test**: `packages/cli/tests/cli.test.ts` — three new cases asserting Dart output,
required-`--target` exit, and invalid-theme rejection.

### Edge Cases

- `currentColor` / unparseable colors → `colorToDart` returns `null`; the emitter skips that
  constant rather than emitting invalid Dart.
- `dimToDart` and `durationToDart` are strict about the structured form (the spec already
  rejects strings); passing a raw string returns `null`.
- Shadow arrays expand inline inside the `<BoxShadow>[…]` literal so the resulting list is
  `const`-constructible.
- Typography `fontWeight` is rounded to the nearest of `100..900` and emitted as
  `FontWeight.w<N>` to match Flutter's enum.

## Requirements *(mandatory)*

- **FR-001**: `transformToDart(theme, options)` MUST validate and resolve the theme via
  `@polymorph/core` before emitting; invalid themes throw the same errors `resolveTheme`
  throws.
- **FR-002**: Output MUST be a single `.dart` file declaring one class with a `static const`
  per resolved semantic + component token, grouped under comment banners by type.
- **FR-003**: A `static ThemeData buildThemeData()` factory MUST be emitted mapping the
  standard semantic colors onto `ColorScheme`, typography onto `TextTheme`, and surface tokens
  onto the matching `ThemeData` fields.
- **FR-004**: Colors MUST accept any CSS Color 4 form via `@polymorph/core.parseColor` and emit
  `Color(0xFFRRGGBB)` (alpha forced to `FF` in v1; non-sRGB converted to sRGB).
- **FR-005**: Dimensions in `rem` MUST be multiplied by 16 before emission; output type is
  Dart `double` (always rendered with a decimal — `8.0`, not `8`).
- **FR-006**: `componentPropDartName(role, property)` MUST flatten dotted roles +
  camelCased properties into one camelCase identifier (`button.primary` + `background` →
  `buttonPrimaryBackground`).
- **FR-007**: `polymorph transform --target dart` MUST be the only invocation surface; no
  in-process API is exposed beyond the package's named exports.
- **FR-008**: The CLI MUST keep its existing exit-code contract: `0` success, `1` invalid
  theme, `2` usage error.
- **FR-009**: Output MUST be deterministic for a given theme + options — manifest iteration
  order, fixed number formatting (no trailing zeros beyond `.0`), no timestamps.

## Assumptions

- v1 is codegen-only; no in-app DTCG runtime. A future spec can layer a runtime loader on top
  if needed (the converters are reusable).
- The opinionated `buildThemeData()` is a starting point — consumers free to ignore it and
  build their own `ThemeData` from the constants.
- Material 3 is the target. A `useMaterial3: true` ThemeData with explicit ColorScheme +
  TextTheme covers what the demos need; cupertino styling is out of scope.
- The mock-bank themes are the de facto fixture surface. Adding new component-token roles to a
  bank will require regenerating goldens (caught by drift guard).
