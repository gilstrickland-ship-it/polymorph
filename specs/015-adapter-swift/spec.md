# Feature Specification: Swift Adapter — SwiftUI Codegen

**Spec ID**: 015-adapter-swift

**Created**: 2026-05-28

**Status**: Implemented

**Input**: Second native (non-web) adapter, mirroring `@polymorph/adapter-flutter`. iOS doesn't
have a JS runtime we can ship a context-provider into, so this adapter is **build-time
codegen**: turn a Polymorph theme into a self-contained `.swift` file that the consumer's Xcode
build compiles. No Xcode toolchain at codegen time — the package emits text.

---

## Overview

`@polymorph/adapter-swift` exports `transformToSwift(theme, options?)` and per-type Swift value
converters; the `polymorph` CLI's existing `transform` command gains `--target swift`.

The emitted file declares two small helper structs (`PolymorphTextStyle`, `PolymorphShadow`)
inlined at the top so the consumer's app has **no external Polymorph dependency** — followed by
a top-level `public enum` (used as a namespace) with one `public static let` per resolved
semantic + component token.

Why an `enum`-as-namespace instead of a `struct` or `class`? Swift enums with no cases cannot
be instantiated, which makes the "this is just static constants" intent explicit and matches
common community conventions (e.g. `enum Constants { ... }`).

## Clarifications

### Session 2026-05-28

- Q: Runtime provider vs. build-time codegen? → A: **Codegen only** for v1, same as Flutter.
  SwiftUI's idiomatic theming is compile-time (`static let` constants, `EnvironmentValues`,
  view modifiers); a runtime DTCG parser would force JSON parsing at app start. Build-time
  emits pure Swift with no Polymorph dependency on the consumer side.
- Q: Helper structs in a runtime package or inlined into the generated file? → A: **Inlined**.
  Avoids forcing the consumer to add a Swift package dependency just to use the constants.
- Q: How to represent typography? → A: SwiftUI's `Font` cannot carry line-height / tracking
  (those are view modifiers), so a small `PolymorphTextStyle` struct bundles everything.
  Consumers apply via a one-line extension (`extension View { func polymorphText(_:) }`).
- Q: How to represent shadows? → A: SwiftUI's `.shadow(color:radius:x:y:)` takes raw values;
  emit `[PolymorphShadow]` (custom struct with `color/x/y/radius`) so single-shadow and
  multi-shadow tokens are uniform.
- Q: Durations in `ms` or `s`? → A: **Swift `TimeInterval`** (seconds, `Double`). `220ms` →
  `0.22`. Animations in SwiftUI all take seconds.
- Q: Cubic-bezier as a custom type or a tuple? → A: **`(Double, Double, Double, Double)`
  tuple** — slots straight into `Animation.timingCurve(c1x, c1y, c2x, c2y, duration:)`.
- Q: Colors — `Color` or `UIColor`? → A: **SwiftUI `Color`** via
  `Color(red:green:blue:)` with `0...1` channels. Apps that need `UIColor` bridge with
  `UIColor(Color(...))`. Alpha forced to 1.0 in v1 (no bank token currently uses translucency).
- Q: Test strategy? → A: **Committed Swift goldens** (`aurora_{light,dark}.swift`,
  `borealis_{light,dark}.swift`) regenerated by `pnpm --filter @polymorph/adapter-swift
  update-goldens` and asserted byte-equal; per-converter unit tests cover edges.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — A SwiftUI app consumes a Polymorph theme (Priority: P1)

A vendor authors `aurora.tokens.json`, runs `polymorph transform aurora.tokens.json --target
swift --mode light --class AuroraThemeLight --output Sources/Polymorph/Theme.swift`, and
imports the resulting enum:

```swift
import SwiftUI

struct ContentView: View {
  var body: some View {
    Text("Hello")
      .foregroundStyle(AuroraThemeLight.colorTextBody)
      .polymorphText(AuroraThemeLight.typographyBody)
      .padding(AuroraThemeLight.spaceMd)
      .background(AuroraThemeLight.colorSurfaceRaised)
  }
}
```

**Independent Test**: `tests/codegen.test.ts` — `transformToSwift(theme, { mode, enumName })`
on the two bank fixtures across `light`/`dark` matches the four committed goldens byte-for-byte.

### User Story 2 — CI catches generator drift (Priority: P1)

`pnpm --filter @polymorph/adapter-swift run update-goldens` regenerates the four `.swift`
files. CI's drift guard re-runs the same script (after building the adapter so `tsx` can
resolve `@polymorph/core` to its `dist/`) and fails on any diff.

### User Story 3 — CLI integration (Priority: P2)

The existing `transform` command grows a second target. Same validate-first behaviour, same
exit codes (`0` success, `1` invalid theme, `2` usage).

### Edge Cases

- `currentColor` / unparseable colors → `colorToSwift` returns `null`; the emitter skips the
  constant rather than emitting invalid Swift.
- `rem` dimensions multiplied by 16 (Swift has no rem; matches the spec's default root size).
- Typography `fontWeight` rounded to the nearest of `100..900` and mapped to SwiftUI's
  `Font.Weight` enum (`.ultraLight`/`.thin`/`.light`/`.regular`/`.medium`/`.semibold`/`.bold`/
  `.heavy`/`.black`).
- Shadow `inset` is dropped with an inline `/* inset not supported by SwiftUI .shadow */`
  comment — SwiftUI's `.shadow` modifier has no inset mode.
- The `--class` flag re-maps to `enumName` for this target (Swift enums are the namespace).

## Requirements *(mandatory)*

- **FR-001**: `transformToSwift(theme, options)` MUST validate and resolve the theme via
  `@polymorph/core` before emitting; invalid themes throw the same errors `resolveTheme`
  throws.
- **FR-002**: Output MUST be a single `.swift` file containing the two helper structs followed
  by one `public enum` namespace with `public static let` per resolved semantic + component
  token, grouped under `// MARK: -` section banners.
- **FR-003**: The helper structs (`PolymorphTextStyle`, `PolymorphShadow`) MUST be inlined in
  the generated file — no external Polymorph runtime package allowed.
- **FR-004**: Colors MUST accept any CSS Color 4 form via `@polymorph/core.parseColor` and emit
  `Color(red: r, green: g, blue: b)` with `0...1` channels (alpha forced to `1.0` in v1).
- **FR-005**: Dimensions in `rem` MUST be multiplied by 16; output is Swift `CGFloat` (rendered
  with an explicit decimal — `8.0`).
- **FR-006**: Durations MUST be emitted as Swift `TimeInterval` (seconds — `220ms` → `0.22`,
  `1s` → `1.0`).
- **FR-007**: `componentPropSwiftName(role, property)` MUST flatten dotted roles + camelCased
  properties into one camelCase identifier (`button.primary` + `background` →
  `buttonPrimaryBackground`).
- **FR-008**: The CLI's `transform` command MUST accept `--target swift` (alongside
  `--target dart`); the `--class` flag is reused to name the Swift enum.
- **FR-009**: Output MUST be deterministic for a given theme + options — manifest iteration
  order, fixed number formatting, no timestamps.

## Assumptions

- v1 is codegen-only; no in-app DTCG runtime. The converters are reusable for a future Swift
  Package-distributed runtime if needed.
- SwiftUI is the primary surface. UIKit consumers can bridge `Color`/`Font` via
  `UIColor(Color(...))` / `UIFont` from the size + weight; sample code in the README.
- The mock-bank themes are the de facto fixture surface. Adding new component-token roles to a
  bank requires regenerating goldens (caught by the drift guard).
- `Font.custom("Inter", size:)` assumes the consumer has registered the custom font in their
  Info.plist; if not, SwiftUI falls back to the system font.
